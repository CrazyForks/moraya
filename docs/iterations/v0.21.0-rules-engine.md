# v0.21.0 — 规则引擎重构（Smart Rules Engine）

**版本：** v0.21.0
**状态：** 已完成
**日期：** 2026-03-03

## 概述

当前 MORAYA.md 规则文件按字符数硬截断注入 system prompt，存在三个问题：内容截断不可预测（可能截在规则中间）、大文件 token 费用不可控、截断后依赖 AI 自行调用工具回读效率低。

本次迭代将规则引擎从「单文件截断」升级为「自动拆分 + 索引按需加载」架构，参考 Claude Code（`.claude/rules/` + frontmatter 条件加载）和 Cursor（`.mdc` + glob 自动挂载）的行业方案，并在此基础上实现**单一源文件自动编译**——用户只维护 `MORAYA.md`，系统自动拆分为 AI 消费的规则片段。

## 设计原则

| 原则 | 说明 |
|------|------|
| 单一事实来源 | 用户只维护 `MORAYA.md`，`.moraya/rules/` 为自动衍生产物 |
| 零学习成本 | 无需了解 glob 语法或多文件管理，写好 Markdown 标题即可 |
| 三方兼顾 | 用户看总纲、AI 读分段、计费按需加载 |
| 自愈合 | `.moraya/` 不提交 git，新机器自动触发重建 |
| 向下兼容 | 无标题结构的 MORAYA.md 降级为整体注入，行为不变 |

## 架构总览

```
MORAYA.md（用户维护的单一源文件）
    │
    ├─ 修改保存
    │
    ▼ sendChatMessage() 触发
DJB2 hash 比对 ─── 无变化 → 使用缓存的拆分结果
    │
    │ 有变化
    ▼
按最高级标题自动拆分
    │
    ├─ 有 <!-- globs: ... --> 标注的段 → 条件加载（A 策略，高优先级）
    ├─ 无 glob 的段 → 索引 + AI 按需 read_file（B 策略）
    └─ 无标题结构 → 降级整体注入 + UI 提示优化建议
    │
    ▼
写入 .moraya/rules/*.md + 更新 .moraya/hash
    │
    ▼
System Prompt = 精简索引 + glob 匹配段内容 + AI read_file 引导
```

## F1: 规则文件自动拆分引擎

### F1.1 拆分触发与缓存

**触发时机**：`sendChatMessage()` 中，在 `buildSystemPrompt()` 之前。

**流程**：

1. 读取 `{sidebarDir}/MORAYA.md` 全文内容
2. 计算 DJB2 hash（复用 `doc-cache.ts` 中已有的 `djb2()` 函数）
3. 读取 `.moraya/hash` 文件（缓存的上次 hash 值）
4. 比对：
   - hash 相同 → 跳过拆分，直接使用 `.moraya/rules/` 下已有文件
   - hash 不同或 `.moraya/` 不存在 → 执行拆分，写入文件，更新 hash
5. 返回拆分结果（索引 + 规则段列表）供 `buildSystemPrompt()` 使用

**缓存目录结构**：

```
{sidebarDir}/
├── MORAYA.md              ← 用户维护（git 提交）
└── .moraya/               ← 自动生成（git 忽略）
    ├── hash               ← MORAYA.md 的 DJB2 hash 值
    └── rules/
        ├── _index.json    ← 拆分索引（标题、文件名、glob、首句摘要）
        ├── code-style.md
        ├── security.md
        └── typescript.md
```

**关键文件**：`src/lib/services/ai/ai-service.ts`

### F1.2 标题层级自适应拆分算法

不硬编码按 `##` 拆分，而是自适应文档实际标题结构：

**算法**：

```
1. 扫描文档，收集所有标题行（# ~ ######）及其级别
2. 找到最高级标题级别 minLevel（如文档用 ## 开头则 minLevel = 2）
3. 如果 minLevel 级别的标题只出现 1 次：
   - 视为文档标题，降级到 minLevel + 1 作为拆分级别
4. 按选定的拆分级别切分文档
   - 每个拆分段包含：标题行 + 该标题下所有内容（含子标题）
   - 拆分级别之上的前导内容（如文档标题、前言）归入第一段或作为 _preamble.md
5. 如果文档完全没有标题 → 降级标记，不拆分
```

**示例 1 — 多个 H2**：

```markdown
# 项目规则           ← 唯一 H1，视为文档标题
## 代码风格          ← 拆分边界 → code-style.md
### TypeScript       ← 属于「代码风格」段
### CSS              ← 属于「代码风格」段
## 安全规范          ← 拆分边界 → security.md
## 部署              ← 拆分边界 → deployment.md
```

**示例 2 — 多个 H1**：

```markdown
# 代码规范           ← 拆分边界 → code-standards.md
## TypeScript        ← 属于「代码规范」段
## Rust              ← 属于「代码规范」段
# AI 配置            ← 拆分边界 → ai-config.md
## 提示词            ← 属于「AI 配置」段
```

**示例 3 — 全部 H3**：

```markdown
### 命名规范         ← minLevel = 3，拆分边界
### 注释规范         ← 拆分边界
### 测试规范         ← 拆分边界
```

**文件命名**：标题文本 → kebab-case（`代码风格` → `code-style.md`，中文标题用 pinyin 或 slug 编号如 `section-01.md`）

### F1.3 Glob 条件标注（A 策略）

用户可在 MORAYA.md 中为特定段落添加 glob 标注：

```markdown
## TypeScript 约定 <!-- globs: **/*.ts, **/*.tsx -->

- 使用 strict mode
- 优先使用 explicit 类型
```

**解析规则**：

- 在标题行的同一行末尾，匹配 `<!-- globs: ... -->` 注释
- 多个 glob 用逗号分隔
- 支持标准 Minimatch 语法（`**/*.ts`, `src/lib/**`, `!**/*.test.ts`）
- 标注后的规则段在 `_index.json` 中记录 `globs` 字段

**注入行为**：

| 场景 | 行为 |
|------|------|
| 有 glob 标注 + 当前编辑文件匹配 | 自动注入该段全文到 system prompt |
| 有 glob 标注 + 当前编辑文件不匹配 | 不注入，不出现在索引中 |
| 无 glob 标注 | 出现在索引中，AI 按需 read_file |

**优先级**：A 策略（glob 匹配的段）优先于 B 策略（索引按需加载）。glob 匹配的段直接注入，不消耗索引的 token 预算。

### F1.4 索引生成（B 策略）

对无 glob 标注的规则段，生成结构化索引：

**`_index.json` 格式**：

```json
{
  "hash": 3948572610,
  "splitLevel": 2,
  "segments": [
    {
      "title": "代码风格",
      "file": "section-01-code-style.md",
      "summary": "TypeScript strict mode、CSS 命名规范、组件结构约定",
      "globs": null
    },
    {
      "title": "TypeScript 约定",
      "file": "section-02-typescript.md",
      "summary": "strict mode、explicit 类型、路径别名",
      "globs": ["**/*.ts", "**/*.tsx"]
    },
    {
      "title": "安全规范",
      "file": "section-03-security.md",
      "summary": "Keychain 存储、CSP 策略、路径验证",
      "globs": null
    }
  ],
  "hasPreamble": true,
  "degraded": false
}
```

**索引摘要生成**：取每段的前 80 个字符（不截断单词/句子中间）作为 summary，不使用 AI 生成摘要（避免额外 token 开销和延迟）。

### F1.5 降级处理

| 场景 | 行为 |
|------|------|
| MORAYA.md 无任何标题 | 整体作为单段注入（兼容现有行为） |
| MORAYA.md 不存在 | 返回 null，与当前行为一致 |
| MORAYA.md 内容为空 | 返回 null |
| `.moraya/` 目录不存在 | 自动创建 |
| `.moraya/hash` 不存在 | 视为 hash 不匹配，触发重建 |

降级时在 `_index.json` 中标记 `"degraded": true`。

## F2: System Prompt 重构

### F2.1 新的 system prompt 注入格式

**替换现有的 `=== MANDATORY — Knowledge Base Rules ===` 整段**。

**新格式**：

```
=== Knowledge Base Rules (MORAYA.md) ===
The user has defined rules for this knowledge base. Follow ALL applicable rules.

[Glob-matched rules — auto-injected based on current file]
--- TypeScript 约定 (auto-loaded: matches **/*.ts) ---
具体规则内容...
--- END ---

[Rules Index — read specific files as needed]
The following rule sections are available. Use read_file to load relevant ones:
1. 代码风格 → {sidebarDir}/.moraya/rules/section-01-code-style.md
   Summary: TypeScript strict mode、CSS 命名规范、组件结构约定
2. 安全规范 → {sidebarDir}/.moraya/rules/section-03-security.md
   Summary: Keychain 存储、CSP 策略、路径验证
Read the relevant sections BEFORE generating content that may be governed by these rules.
=== END Rules ===
```

### F2.2 Token 预算控制

| 组成部分 | 预算 |
|---------|------|
| Glob 匹配段（自动注入） | 无上限（用户通过 glob 标注显式控制） |
| 索引部分 | 最大 2000 tokens（约 100 行索引） |
| 前言段（_preamble.md，如有） | 最大 500 tokens |

当索引超出预算时：只列标题 + 文件路径，省略 summary。

### F2.3 移除旧的截断逻辑

- 删除 `readMorayaMd()` 中的 `aiRulesMaxChars` 截断逻辑
- 删除 settings-store.ts 中的 `aiRulesMaxChars` 字段
- 删除 SettingsPanel.svelte 中对应的 UI 控件
- 删除 12 个 locale 文件中的 `settings.permissions.aiRulesMaxChars` 和 `settings.permissions.aiRulesMaxCharsHint` 键
- 删除 system prompt 中的 `[TRUNCATED]` 提示和 `WARNING: Rules were truncated` 文案

### F2.4 MORAYA.md Convention 指令更新

更新 system prompt 中的 MORAYA.md Convention 段，增加结构化规则：

```
--- MORAYA.md Convention ---
This editor supports a per-knowledge-base AI rules file called MORAYA.md.

When creating or modifying MORAYA.md:
- The file MUST be named exactly MORAYA.md, placed at knowledge base root: {sidebarDir}/MORAYA.md
- Use ## headings to organize rules into sections (each section becomes independently loadable)
- Optional: add <!-- globs: pattern --> after a heading to scope rules to specific file types
  Example: ## CSS Rules <!-- globs: **/*.css, **/*.scss -->
- Keep each section focused on one concern for optimal rule loading
- The system automatically splits MORAYA.md into indexed segments for efficient AI consumption
--- End MORAYA.md Convention ---
```

## F4: `.moraya/` 自愈合与 Git 集成

### F4.1 自动创建 `.moraya/` 目录

在拆分引擎首次运行时，如果 `.moraya/` 目录不存在，自动创建：

```
{sidebarDir}/.moraya/
{sidebarDir}/.moraya/rules/
```

通过 Tauri `invoke('write_file')` 写入文件时自动创建父目录。

### F4.2 自愈合机制

`.moraya/` 不提交 git，因此：

- 新机器 clone 仓库 → `.moraya/` 不存在 → hash 比对必然失败 → 自动触发拆分重建
- 用户手动删除 `.moraya/` → 下次 AI 对话时自动重建
- `MORAYA.md` 被其他工具修改 → hash 变化 → 自动重新拆分

### F4.3 自动追加 `.gitignore`

当规则引擎首次创建 `.moraya/` 目录时：

1. 检查 `{sidebarDir}/.gitignore` 是否存在
2. 如果存在，检查是否已包含 `.moraya/` 条目
3. 如果不包含，追加 `.moraya/` 到 `.gitignore` 末尾
4. 如果 `.gitignore` 不存在，不创建（用户可能不使用 git）
